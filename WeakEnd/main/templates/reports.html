Local, Remote File Inclusion

취약점 요약
PHP에서 include라는 함수 등을 이용하여 다른 파일을 소스코드에 포함시켜 원하는 정보를 유출시키는 공격할 수 있는 취약점이다.

취약점 설명
주로 PHP 애플리케이션을 대상으로 include라는 함수를 사용하여 다른 파일을 소스코드에 직접 포함시킬 수 있는 취약점이다. include 할 수 있는 파일의 위치가 호스트 내부/외부인지에 따라 Local File Inclusion/Remote File Inclusion으로 구분된다. 호스트 외부의 파일을 포함시킬 수 있는 RFI가 더 심각한 공격으로 여겨진다.
공격은 http://localhost/dvwa/vulnerabilities/fi/?page=bad.php 와 같이 get 방식으로 파일을 불러오거나 post 방식으로 파일을 불러올 때 파라미터를 호스트 내/외부에 존재하는 다른 파일로 지정하여 발생한다.

해결 방안
- 외부 사용자가 입력한 파일 이름을 include에 사용하지 않는다.
- 블랙리스트 방식을 사용하여 파일 경로를 입력 받을 때 “http://”, “https://”, “../” 등의 문자열을 필터링하여 RFI, LFI 공격을 대비한다.
- 화이트리스트 방식을 사용하여 존재하는 파일을 리스트로 만들어 해당 파일들만 허용한다.

취약점 요약
시스템 명령어를 쿼리문에 주입하여 취약한 변수를 통해 서버 운영체제에 접근하는 공격을 할 수 있는 취약점이다.

취약점 설명
웹 애플리케이션에서 사용자가 입력한 값을 입력 값 검증 없이 시스템 명령어의 일부분으로 전달하여 발생한다. 공격자는 이 필드에 “&”, “&&”, “;”, “|”, “||” 등을 사용하여 “& netstat -an” 등의 원하는 시스템 명령어를 실행할 수 있다. 공격에 성공한다면 매우 큰 피해를 입힐 수 있다.
취약 함수 목록
Java: System.*, 특히 System.runtime 취약, Runtime.exec()
C/C++: system(), exec(), ShellExecute()
python: exec(), eval(), os.system(), os.popen(), subprocess.popen(), subprocess.call()
Perl: open(), sysopen(), system(), glob()
php: exec(), system(), passthru(), popen(), rquire(), include(), eval(), preg_replace(), shell_exec(), proc_open(), eval()

해결 방안
- 직접적으로 시스템 명령어를 호출하지 않는다.
  예) system(“mkdir $dir_name”) -> mkdir($dir_name)
  system("cat /$something") -> system("cat", "/$something")
- 시스템 명령어를 사용할 수밖에 없다면 입력 값 검증을 거친다.

취약점 명
SQL Injection

해당 사이트 취약점
SQL Injection Vuln Points
{{ json_data.SQLI}}

취약점 요약
입력 폼에서 입력 값 검증을 하지 않아 입력 값으로 SQL 구문을 입력하여 DB의 정보를 열람하거나 조작할 수 있는 취약점이다.

취약점 설명
펜타시큐리티의 2016년 웹 공격 동향 보고서에 따르면 SQL Injection은 전체 해킹 시도 중 절반에 가까운 비중을 차지한다. SQL Injection은 웹 애플리케이션이 데이터베이스와 연동하는 모델에서 발생 가능하다. 입력 폼에서 사용자에게 입력을 받아 데이터베이스로 전달할 때 입력 값을 검증하지 않아 조작된 SQL 구문이 데이터베이스에 전달되면서 의도하지 않은 동작을 하며 발생한다. 간단한 예로 사용자가 SELECT * FROM Users WHERE id = ’INPUT1’ AND password = ‘INPUT2’를 사용하는 입력 폼의 INPUT1에 ‘ OR 1=1 – 과 같은 값을 입력한다면 SELECT * FROM Users가 되어 Users 테이블에 있는 모든 정보를 조회할 수 있게 된다.
공격 기법 목록
- 쿼리 조건 무력화(Where 구문 우회) Where 조건문이 무조건 참(True)이 되도록 하는 공격이다. 로그인 페이지를 예로 (Select * From Users Where UserID = 'admin'-- And Password = '아무거나') 와 같이 원하는 부분을 주석처리 함으로써 Where 조건을 무력화하여 패스워드를 몰라도 인증을 통과하는 방법이 있다. 다른 방식으로는 (Select * From Users Where UserID = 'test' And Password='1234' or '1'='1') 와 같이 or 을 사용해서 Where 조건문이 무조건 참(True)이 되도록 하는 방식, (Select * From Users Where UserID = 'admin' ; DELETE From Users -- And Password='아무거나') 와 같이 ;(세미콜론)을 사용하여 한 줄로 된 두개의 명령을 실행 시키도록 하는 방식이 존재한다. - 고의적 에러 유발 고의적으로 SQL 구문 에러를 발생시켜 웹 애플리케이션이 반환하는 오류 정보를 이용해 유용한 정보를 알아차리는 공격이다. 기본적으로 웹 애플리케이션은 쿼리 수행 중 오류가 발생하면 DB 오류를 그대로 브라우저에 출력하는데, 이를 통해 DB의 스키마 정보나 데이터가 유출될 수 있다. 주로 또 다른 공격의 소재로 사용하기 위해 사용하는 방식이다. (Select * From Users Where UserID = 'test' UNION SELECT 1 -- And Password='아무거나')를 예로 컬럼 개수가 일치하는 두 테이블의 결과를 합치는 Union 명령을 사용하는 경우 컬럼의 개수가 일치하지 않으면 다음과 같은 결과를 반환한다. "UNION, INTERSECT 또는 EXCEPT 연산자를 사용하여 결합된 모든 쿼리의 대상 목록에는 동일한 개수의 식이 있어야 합니다." (SQL Server 기준) 그럼 공격자는 컬럼의 개수를 늘려가며 공격을 하여 데이터의 형태를 알아낼 수 있다. - 시스템 명령어 실행 MS SQL Server의 경우 시스템 명령을 실행할 수 있는 확장 프로시저를 제공한다. (Select * From Users Where UserID = 'admin' ; EXEC master.dbo.xp_cmdshell 'cmd.exe dir c:'-- And Password='아무거나')를 예로 공격자는 C 드라이브를 탐색하는 명령을 보낼 수 있다. 해당 계정에 유효한 권한이 주어져 있다면 어떤 시스템 명령도 내릴 수 있게 된다. - Boolean-based blind 공격 만일 공격 대상 웹페이지가 어떠한 오류도 출력하지 않고 쿼리 결과 리스트도 제공하지 않는다면 위의 세가지 방법으로는 공격에 성공하기 쉽지 않다. 이 경우 Blind SQL Injection을 통해 공격을 시도한다. 해당 공격을 위해선 쿼리 결과에 대한 참/거짓을 반환하는 ID 찾기나 게시판 검색과 같은 기능을 사용한다. 게시판 검색 기능을 예로 들어 임의의 검색어와 함께 And 연산자를 이용해 특정 구문을 입력할 경우 참인 구문을 입력하면 게시판 검색이 작동하고 거짓인 구문을 입력하면 게시판 검색이 작동하지 않는다. 이러한 결과를 사용해서 공격을 시도한다. - Time-based Blind 공격 만약 boolean-based blind 공격에서 참/거짓이 모두 응답이 동일한 경우 시간을 지연시키는 쿼리를 이용해서 조건문의 참/거짓을 판별할 수 있다. MS SQL Server의 경우를 예로 들어 (SELECT * FROM TB_Boards WHERE Title = 'hello' ; IF SYSTEM_USER='sa' WAITFOR DELAY '00:00:5') 와 같은 쿼리를 구성하였을 때, 응답이 5초간 지연된다면 시스템 계정이 “sa”라고 판단할 수 있다. 공격자는 이와 같이 쿼리 조건 무력화, 고의적 에러, Boolean-based, Time-based 공격 후 정보획득, 시스템 명령어 실행, 인증을 우회, 관리자와 다른 사용자의 계정 탈취 등의 공격을 수행한다.
해결 방안
- Prepared Statement 구문을 이용하여 DB에 컴파일 된 쿼리문(문자열)을 전달하는 방법을 사용한다. - Parameterized Statement를 사용하는 경우, 외부 입력데이터에 대하여 특수문자 및 쿼리 예약어를 필터링한다. - DB 계정을 분리하고 DB 계정의 권한을 최소화한다. - DB의 기본/확장 프로시저를 제거한다. - Struts, Spring 등과 같은 프레임워크를 사용하는 경우, 외부 입력 값 검증 모듈을 사용한다.

취약점 명
Cross Site Scripting(XSS)

해당 사이트 취약점
Cross Site Scripting Vuln Points
{{ json_data.XSS}}

취약점 요약
악의적인 동작을 하는 스크립트를 삽입해 다른 사용자에게 전송하는 취약점이다.

취약점 설명
공격자가 웹 페이지에 스크립트(주로 Javascript)를 삽입하여 의도치 않은 동작을 수행시키거나, 쿠키 혹은 세션 토큰과 같은 민감한 정보를 탈취하는데 사용한다. 여러 사용자가 사용하는 게시판 등에 코드를 삽입하거나, 메일 등에 악성 URL을 포함시키는 방식으로 전파된다.
공격 기법 종류
- 스크립트 태그 방식 가장 기초적인 방식으로 Script 태그를 사용하는 방식 - 자바스크립트 링크 방식 링크를 클릭하면 스크립트가 실행되는 방식 - 이벤트 속성 방식 onload, onerror,onclick 등의 이벤트 속성을 사용하는 방식 - 블랙리스트 우회 방식 알려지지 않은 태그와 속성을 사용해 블랙리스트를 우회하는 방식 - 내용 난독화 방식 HTML 인코드를 사용하여 스크립트를 난독화 한 뒤 삽입한다.

위 기법들을 사용한 공격은 3종류로 나눌 수 있다. - 저장 XSS 공격 웹 사이트의 게시판, 코멘트 등에 악성 스크립트를 영구 저장해 공격하는 방식이다. 다른 사용자가 악성 스크립트가 저장된 페이지에 정보를 요청할 때 공격이 수행된다.
- 반사 XSS 공격 사용자가 서버로 입력한 값을 서버가 요청한 사용자의 브라우저로 악성스크립트를 반사시키는 공격이다. 주로 악성 스크립트가 포함된 URL을 클릭하도록 유도하여 클릭 시 사용자를 공격하는 방식이다.
- DOM 기반 XSS 공격 DOM 기반 XSS는 원본 클라이언트 측에서 사용하는 희생자의 브라우저에서 DOM "환경"을 수정 한 결과 공격 페이로드가 실행되는 XSS 공격이다 스크립트를 사용하여 클라이언트 측 코드가 "예기치 않은"방식으로 실행되도록 한다. 즉, 페이지 자체 (HTTP 응답은 변경되지 않음)가 변경되지 않지만 DOM 환경에서 발생한 악의적 인 수정으로 인해 페이지에 포함된 클라이언트 측 코드가 다르게 실행된다.
공격자는 XSS 공격을 통해 쿠키정보/세션ID 등의 정보를 획득하거나, 시스템 관리자 권한을 획득하거나, 악성코드를 다운로드 하도록 유도한다.
해결 방안
- 입출력 값 검증   입출력 값에 화이트리스트 방식으로 일부 태그와 속성만 허용하도록 하고 특수문자를 필터링 혹은 대체하여 공격을 예방한다. - XSS 방어 라이브러리 사용   보안 업체 혹은 기업에서 만든 라이브러리 사용 - 웹 방화벽 사용
